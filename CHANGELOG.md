# Changelog
## [0.25] - To be announced
*Measures and community strategies gain per-instance parameters and a shared drag-and-drop builder. The path-, flow-, and brokerage-based measures, attacks, and community detectors are removed. SBM new community strategy.*

### New features
- **New community strategy: stochastic block model (`SBM`).** A directed, degree-corrected Bayesian SBM (Karrer & Newman 2011; Peixoto 2014/2017), fitted by minimum description length via `graph-tool`. Unlike the modularity family, which only finds *assortative* communities, the SBM recovers arbitrary block structure — disassortative groups, core-periphery layers, and the bipartite source/amplifier pattern that dominates a forward-attribution network. Its blocks are **citation-role / structural-equivalence classes** (Lorrain & White 1971), not necessarily cohesive groups: two channels share a block when they cite, and are cited by, the rest of the network the same way — which keeps it consistent with Pulpit's one-degree attribution model (the block-affinity matrix is a one-step group-to-group citation rate, not a flow). It is degree-corrected so the partition reflects structure *beyond* the in-degree heterogeneity of the star topology, unweighted, and seeded for reproducibility. Parameterised by `mode` (`NESTED`, the default, takes the finest level of the nested SBM for robust model selection on large graphs; `FLAT` fits a single level) and repeatable per mode, like `LEIDEN_CPM`. **`graph-tool` is not installable from pip** — install it via conda-forge or your system package manager; if it is absent, requesting `SBM` fails with a clear message and the other strategies are unaffected. See `docs/community-detection.md`.
- **Measures are selected by drag-and-drop, and parameterised ones can be added more than once with different parameters.** The Operations panel's measure checkboxes become a palette → ordered drop-zone; `DIFFUSIONLAG` and `MODULEROLE` carry inline parameters and may repeat (e.g. `DIFFUSIONLAG(window=30)` alongside `DIFFUSIONLAG(window=90)`), each producing its own parameter-suffixed output column. On the CLI, `--measures` gains keyword tokens — `DIFFUSIONLAG(window=90)`, `MODULEROLE(basis=LEIDEN_DIRECTED)`, etc. The community basis is chosen per instance.
- **Community strategies now use the same drag-and-drop builder as measures, with per-instance parameters.** The Operations panel's community-strategy checkboxes become a palette → ordered drop-zone (the measures builder, factored into a shared component). `LEIDEN_CPM` carries an inline parameter — resolution γ — and may be added more than once, e.g. a multi-resolution scan `LEIDEN_CPM(resolution=0.01)` alongside `LEIDEN_CPM(resolution=0.05)`, each producing its own parameter-suffixed partition column. This folds the old fixed `LEIDEN_CPM_COARSE` / `LEIDEN_CPM_FINE` presets into one parameterised `LEIDEN_CPM`. On the CLI, `--community-strategies` gains keyword tokens (`LEIDEN_CPM(resolution=0.05)`) and `--leiden-coarse-resolution` / `--leiden-fine-resolution` are replaced by `--leiden-cpm-resolution` (old flags kept as deprecated aliases for one release). Measures and community strategies share one token engine (`network/tokens.py`) and one JS builder, so the two stay in lock-step.
- **The project title now shows in the web app, not just the exports.** The Manage › Project title appears in the browser-tab `<title>` and the About modal, so analysts running several instances can tell them apart at a glance; it previously surfaced only in the HTML/XLSX exports. The login page stays generic so the project name isn't shown to anonymous visitors in `PROTECTED` mode.
- **Edge weight can be shown as line thickness in the 2D and 3D maps.** A new **Show edge weight** toggle (Options › Styles, off by default) scales each edge's width to its citation weight via a shared log-scaled, p2–p98-clamped mapping that stays legible at any `--edge-weight-strategy` and degrades to a uniform width when there is no spread; the 3D viewer uses GPU "fat lines" since plain WebGL lines can't vary width.
- **Two-way connections render as two distinct edges, with an optional arrowhead toggle, in the 2D and 3D maps.** The graph is directed (citing → cited), and a mutual tie already carries two edges with their own weights; both maps now draw a one-way citation as a straight line and a two-way one as a pair of arcs bowing to opposite sides, so each direction reads as its own edge — and, with **Show edge weight** on, its own thickness. A new **Show arrows** toggle (Options › Styles, off by default) adds an arrowhead at each edge's target to make direction explicit; the 3D viewer draws them as instanced cones that track the edges through selection, year, and layout changes.
- **The path-, flow-, and brokerage-based measures and attacks are removed.** Telegram attributes every forward to the original source, so Pulpit's graph is a one-degree citation network, not a transmission network: its multi-hop paths carry no flow (Borgatti 2005), and a literature review found no peer-reviewed validation of these measures on Telegram or other original-source-attribution networks. Nine measures are gone — **betweenness**, **harmonic centrality**, **k-core coreness**, **collective influence**, **trophic level**, **Gould-Fernandez brokerage roles**, **SIR spreading efficiency**, **Hwang bridging centrality**, and **community bridging** — along with five robustness attack orders (`betweenness`, `betweenness_dyn`, `harmonic`, `bridging`, `spreading`) and the `--spreading-runs` flag / robustness bridging-basis selector. The surviving battery is the radial/degree family (PageRank, HITS, in/out-degree), the local-structure measures (Burt's constraint, local clustering), the Guimerà-Amaral within-module role, and the behavioural measures (amplification, content originality, diffusion lag). The same one-degree rationale removes the **flow / random-walk community detectors** — `INFOMAP`, `INFOMAP_MEMORY`, `MCL`, `WALKTRAP`, `STRONGCC` (and the `--mcl-inflation` flag) — since modularity is a single-step walk while those integrate many steps (Lambiotte-Delvenne-Barahona 2014); the density/structural detectors `ORGANIZATION`, `LEIDEN`, `LEIDEN_DIRECTED`, `LEIDEN_CPM`, `LABELPROPAGATION`, and `KCORE` remain. Existing `.operations-structural` files are auto-migrated (removed measure / attack / strategy tokens stripped on load). `docs/network-measures.md` cites the basis for the cut — Kuznetsov et al. (2026) for the empirically depth-1 Telegram forwarding cascades, Borgatti & Everett (2006) for Gould-Fernandez brokerage being a 2-betweenness variant, and Liu et al. (2015) for the independent k-core super-spreader failure.

### Fixes
- **Dead-leaf nodes could hang in the graph with no edges.** A dead leaf earns its slot from the all-time, period-blind `Channel.in_degree`, but edge construction is period-aware — a citation only counts while the citing channel is in an in-target period. The orphan-pruning sweep that drops dead leaves left edgeless only ran for windowed exports, so an all-time run kept dead leaves whose only citations fell outside the citing channels' in-target periods as isolated ghost nodes (~85 in a representative export). The sweep now runs whenever dead leaves are drawn; isolated *in-target* channels are still kept, since they are subjects of the analysis.
- **Vacancy "Brokerage overlap" (Score C) reinterpreted for the one-degree model.** The score is unchanged — still the Jaccard overlap of the (source-org, amplifier-org) pairs a candidate spans — but its wording no longer claims the channel *mediates flow* between those organisations: under one-degree attribution the orgs it forwards from and the orgs that amplify it are unrelated citation streams, so the metric now reads as a structural-*position* overlap (same source and audience ecosystems), not content routed through the channel. Docstrings, the Operations tooltip, and the vacancy-analysis docs updated; no behaviour change.
- **Dropped four unused Python dependencies** from `requirements.txt`: `infomap` and `markov-clustering` (no longer imported once the flow-based community detectors were removed), and `beautifulsoup4` / `lxml` (already unreferenced anywhere in the codebase).
- **A search term saved to the database from the Operations panel was searched twice** — checking "Save to database" both persisted the term and passed it as `--extra-term`, so `search_channels` hit it once as a database term and again as an extra term; the command now skips an extra term already searched as a database term in the same run.
- **Interest-structural export always labelled windowed runs "windowed".** `_scope_label` read `date__gte`/`date__lte`, but the export builds the window with the `__date` transform (`date__date__gte`/`lte`), so the `structural_scope` field never showed the actual date range — it now reads the production keys (plain form kept as a fallback).
- **Test-suite output denoised.** `manage.py test` buried its result under ~180 lines of management-command progress, logger records from deliberately-exercised error paths, a data migration's prints, naive-datetime warnings, and umap's Tensorflow `ImportWarning`; all are now silenced under the test runner only (production logging and command output unchanged).

## [0.24] - 2026-06-01
*Further consolidation of measures.*

### New features
- **Edge direction is no longer a knob — and two measures now reverse internally where the academic definition requires it.** The `REVERSED_EDGES` Django setting and the `[graph].reversed_edges` TOML key are gone; the graph is always built in the citation convention (a forward of source Y's content by amplifier X produces an X→Y edge, citing→cited), so `Channel.in_degree` is unambiguously "times cited" and `out_degree` is "times citing".
- **Network-measure overhaul: dropped four, added three.** The per-channel battery is retuned for the directed, sparse, partially disconnected citation graphs Pulpit builds. **Removed** (lowest marginal value for this graph type, together with their matching robustness attacks `katz` / `closeness` / `flow_betweenness` / `katz_dyn`): **Closeness centrality** (`CLOSENESS`) — dominated by Harmonic centrality, the reciprocal-distance reachability measure built for disconnected graphs; **Flow betweenness** (`FLOWBETWEENNESS`) — symmetrised the graph to undirected and zeroed every non-giant-component node, discarding the edge direction that is the whole point of a citation graph; **Katz centrality** (`KATZ`) — an eigenvector-prestige sibling of PageRank carrying an α-stability tax; and **Ego network density** (`EGODENSITY`) — a near-duplicate of Local clustering. **Added**: **K-core coreness** (`CORENESS`) — the deepest k-core a channel survives in (Kitsak et al. 2010), a robust spreading-influence predictor, computed on the symmetrised graph like the KCORE community strategy; **Trophic level** (`TROPHICLEVEL`) — a structural source→sink coordinate (MacKay, Johnson & Sansom 2020 hierarchical levels, finite on cyclic graphs), the structural counterpart of content originality; and **Within-module role** (`MODULEROLE`) — the Guimerà & Amaral (2005) node-role taxonomy, emitting the numeric within-module degree z-score (`within_module_z`) plus the categorical 7-role `module_role` label over the community partition (basis auto-resolved from the labels already on the nodes, defaulting to Leiden Directed; `module_role` is exported as a string column like `organization`). Net per-channel measure count is unchanged at 18.
- **Two academically-grounded measures added — per-channel battery now 20.** **Collective Influence** (`COLLECTIVEINFLUENCE`) — the optimal-percolation key-spreader score of Morone & Makse (2015, *Nature*): `(k−1) · Σ(k−1)` over the nodes exactly two hops out, on the symmetrised unweighted graph (like `CORENESS`). It ranks the channels whose removal most fragments the network — better than degree, k-core, PageRank or betweenness — and is the per-node dual of the robustness dismantling order. **Brokerage roles** (`BROKERAGEROLES`) — the Gould & Fernandez (1989) brokerage census: every directed citation 2-path a channel mediates is typed by the groups of its endpoints into coordinator / gatekeeper / representative / consultant / liaison, reported as a sortable `brokerage_total` plus a dominant-role label over the Organization partition (its preferred basis), with the five raw role counts written to `nodes.csv` and the GEXF/GraphML exports. It is the *typed* complement to Burt's constraint and community bridging, which measure brokerage intensity without naming the kind. Both are opt-in via `--measures` or the Operations panel; `BROKERAGEROLES` needs a community strategy.
- **Behavioural equivalence matrix** — new opt-in page (`--behavioural-equivalence`, or the Operations panel checkbox) showing pairwise cosine similarity of channels' *behavioural-measure* profiles: amplification, content originality, diffusion lag, spreading efficiency, and audience/activity volume (followers, message count). Min-max normalised per measure, missing values imputed to the column median. It is the behavioural counterpart to the (now correct) structural equivalence matrix — two channels can be structurally equivalent yet behave very differently.
- **Network-measure docs rewritten for non-technical readers.** All 18 sections in `docs/network-measures.md` consolidated into a single uniform structure — one-line plain-language summary, short conceptual description (no formulas, no implementation details), 1–2 essential references, an **In practice** read against Pulpit's threat-mapping goals, and a worked **Example** vignette written for journalists, activists, and other readers who care about the outcome rather than the maths. Length harmonised to ~250–380 words per measure, so the file now reads as a coherent guide rather than 18 variably-deep academic write-ups. Supersedes the per-measure long-form rewrites that landed across earlier 0.24 commits.
- **Project dossier moves from `.env` to the Manage panel.** The project title — previously the `PROJECT_TITLE` environment variable, baked into the `<title>` of `structural_analysis` HTML/XLSX exports — is now a database singleton edited at **Manage › Project**, alongside three free-text fields: description, criteria, and notes. On upgrade the existing `PROJECT_TITLE` value is migrated into the new record; the `.env` key and the `PROJECT_TITLE` Django setting are removed.
- **Python 3.12, 3.13, and 3.14 are all fully supported.** `setup.sh` and `setup.bat` now accept any of the three interpreters (and `setup.sh` prefers the newest installed) rather than pinning 3.12 exactly; the getting-started guide states the supported range. `ruff` keeps targeting the 3.12 floor so lint and formatting stay compatible across the whole range.

### Fixes
- **Vacancy succession catalog trimmed from 6 measures to 4; Temporal Adoption reformulated.** Dropped `CASCADE_OVERLAP` (Jaccard of SIR diffusion reach sets) and `PPR` (Personalized PageRank seeded on the orphaned amplifiers). On Pulpit's directed citation graph both were dominated by the structural triad already in the default set — Amplifier Coverage + Neighbour-set Equivalence + Brokerage overlap collectively answer "same audience, same neighbourhood, same brokerage role" without the Monte Carlo SIR cost of CASCADE_OVERLAP or the upstream-supply-chain framing of PPR that mostly recapitulated what coverage already surfaced. Temporal Adoption switched from "coverage × recency decay" to a hyperbolic discount `coverage / (1 + mean_days_to_adoption / 30)` (Mazur 1987), which scores faster-adopted candidates higher than slow ones while remaining well-defined when adoption is bimodal (by Jensen's inequality on the convex `1/(1+x)`, mean-then-discount is more pessimistic than discount-then-mean on mixed fast+slow adopter sets). The `--vacancy-ppr-alpha` flag, the PPR α numeric input in Operations, and the `vacancy.ppr_alpha` settings/config key are removed. Existing `.operations-structural` files that list `CASCADE_OVERLAP` / `PPR` under `[vacancy].measures` need those tokens removed.
- **Community strategy catalog trimmed from 13 to 11.** Dropped `LOUVAIN` and `WEAKCC`. Louvain is strictly superseded by Leiden (same modularity null model, plus a connectivity guarantee that fixes the stranded-node failure mode); on Pulpit's typical graph sizes Leiden's runtime is fine, so keeping both was carrying a legacy baseline with no analytical lift. WEAKCC was never a community detector — the docs already said so and the consensus / NMI matrices already excluded it — and the disconnection diagnostic it provided is already covered by the WCC headline stats in network-stats. UI checkboxes, CLI tokens, docs sections, and the cross-references in the remaining strategy entries all updated. Existing `.operations-structural` files that still list `LOUVAIN` or `WEAKCC` under `[communities].strategies` need those entries removed — `structural_analysis` now rejects them as invalid.
- **Robustness strategy catalog trimmed from 17 to 12.** Dropped five attacks that added little to no signal over what the remaining catalog already covers on Pulpit's directed citation graph: `hits_hub` and `hits_authority` (plus their dynamic twins `hits_hub_dyn` / `hits_authority_dyn`) — in this orientation they collapse to amplified out-strength and in-strength respectively, and the HITS scorers already fall back to those strengths when convergence fails; and `burt_constraint`, whose `nx.constraint` symmetrises the directed graph internally (see the matching note above), making it the only brokerage attack that is direction-blind on a citation graph whose whole story is the amplifier→source direction. The HITS hub/authority and Burt's constraint **node measures** are unchanged — they remain in the channel table, the centrality registry, and the network-stats summary. Only the robustness strategies and their UI checkboxes / CLI tokens / docs are removed.
- **Authority-weighted reach (D) fallback was mis-documented** — `interesting-messages.md` said D fell back from PageRank → HITS Authority to "in-degree centrality", but the code falls back to **in-strength** (`in_deg`, the weighted in-degree always populated by `apply_base_node_measures`), which is the closer match to Cha et al. 2010's "weighted indegree" that D adapts. Doc now matches code.
- **Harmonic centrality docs had the direction inverted** — every prose description (network-measures.md, README, CLAUDE.md, robustness-analysis.md, channel-table and Operations-panel tooltips) said harmonic measures *outgoing* reach ("which channels can reach the rest of the network"), but the code calls `nx.harmonic_centrality` on the as-built citation graph, which computes `Σ_v 1/d(v, u)` — the *incoming* form (how easily the rest of the network reaches `u`). Implementation is unchanged (academically valid: in-harmonic with `distance = 1/weight` per Opsahl 2010, normalised by `n−1` per Boldi & Vigna 2014); all prose now matches the code.
- **Vacancy analysis ignored attribution periods** — its message queries (orphaned-amplifier detection, candidate selection, and the amplifier-coverage / structural-equivalence / temporal / cascade scorers, in both the interactive card and the `structural_analysis` export) filtered by date only and treated any *ever*-in-target channel as in-target, so a forward sent while a channel was out of its in-target period still counted — unlike the rest of the pipeline, whose single chokepoint `channel_cutoff_q()` requires the message's channel to be in-target *at the message date*. All those queries now apply `channel_cutoff_q()`, so vacancy scores match the graph and the period-aware measures (channels with out-of-target stretches score slightly lower; always-in-target channels are unchanged).
- **`structural_analysis` from the bare CLI crashed before building the graph** — its option resolver fell back to hardcoded empty strings instead of the config/built-in defaults that the Operations panel supplies as explicit flags, so a flag-less run resolved `channel_types` to `[]` (matched no channels) and `edge_weight_strategy` to `""` (the PARTIAL_REFERENCES branch only fills the normaliser for the exact token, so every edge weight became 0) — both surfacing as the misleading "There are no relationships between channels" — and `community_palette` to `""` (`Palette '' not found`). All three now fall back to the config value, then to the documented built-in default (`DEFAULT_CHANNEL_TYPES`, `PARTIAL_REFERENCES`, vaporwave-reversed); an explicit flag still wins and the Operations panel path is unchanged.
- **"Structural similarity" matrix wasn't structural equivalence** — it computed the cosine of channels' *centrality fingerprints* (a feature vector of all network measures, including followers and message count, with missing values forced to 0), so two channels could score 1.0 while sharing no neighbours. The page now computes genuine **structural equivalence** (Lorrain & White 1971): the cosine of each channel's weighted in + out *tie profile* (`P = [A | Aᵀ]`), so a high score means two channels cite — and are cited by — the same channels. The page is retitled "Structural equivalence" (the `structural_similarity.html`/`.json` filenames and `--structural-similarity` flag are kept for saved-config and URL compatibility); the old centrality-fingerprint method moves, properly scoped, into the new behavioural equivalence matrix above.
- **Burt's constraint was labelled and documented as direction-sensitive** — it is not. `nx.constraint` symmetrises the directed graph internally (`mutual_weight(u, v) = w(u→v) + w(v→u)`, `N(v) = predecessors(v) ∪ successors(v)`), matching Burt's framework where structural holes are about ego's contacts regardless of citation direction. Implementation is unchanged (academically valid: Burt 1992, 2004). Operations panel arrow icon now reads "Undirectional — graph symmetrised"; the harmonic-centrality docstring and CLAUDE.md no longer list Burt's constraint among the as-built-direction measures. Typical range stays [0, 1] with a noted theoretical upper bound ≈ 1.125 (Burt 1992 ch. 2; confirmed empirically — unweighted complete triangle gives 1.125 exactly).

## [0.23] - 2026-05-27
*Time-bounded organization attribution. Fixes, lots and important ones.*

### New features
- **Time-bounded organization attribution** — a channel's membership in an `Organization` is no longer a static `Channel.organization` FK plus the one-way `out_of_target_after` cutoff; it is now a timeline of non-overlapping **`ChannelAttribution`** periods (`channel` + `organization` + optional inclusive `start`/`end`, both `None` meaning "since creation" / "until now"). A channel can therefore be in-target only for a stretch of time, or in-target across a long span under *alternating* organizations, and a message or relationship counts as in-target only when its date falls inside one of its channel's in-target periods. Non-overlap is enforced in `ChannelAttribution.clean()`, the DRF serializer, and the admin inline formset (SQLite can't express a portable exclusion constraint). Migrations `0047`–`0049` create the model, backfill one period `(start=None, end=out_of_target_after)` per channel that had an organization, then drop the two legacy `Channel` fields.
  - **Analysis is period- and window-aware.** The single chokepoint `network/utils.channel_cutoff_q()` becomes a period-aware `Q(Exists(ChannelAttribution …))` (with the cheap single-channel `channel_period_date_q()` for per-channel views), so graph edges, degrees, measures, stats, and the home summary all honour attribution periods. The graph is still **one node per channel**; its representative organization — used for the ORGANIZATION community grouping, node colour, and the new **Organization** column in the channel table / `nodes.csv` / GEXF / GraphML — is the in-target org whose period covers the **most days inside the analysis window**, tie-broken by earliest start (`network/graph_builder.resolve_window_organization`).
  - **Crawling.** `crawl_channels` selects channels that have *ever* been in-target (any in-target period) plus `to_inspect` channels; message storage is gated to in-target periods (generalising the old `out_of_target_after` skip), while `to_inspect` channels are still crawled in full. The hole-fixer is now period-aware — a gap whose bounding stored messages both fall outside the channel's in-target periods is intentional and is no longer re-fetched on every run.
  - **Vacancy analysis** resolves the organization of each forwarded-from channel **as of the forwarding message's date** (via `ChannelAttribution.build_cache`/`org_at`), so brokerage scoring reflects who a channel belonged to at the time, not today.
  - **UI.** The `/manage/` channel editor gains an **Attribution periods** editor (add/remove rows of organization + start + end, with client- and server-side overlap validation, saved through the new `channel-attributions` API); the channels list shows a read-only **current organization** column; bulk-assign now replaces a channel's whole attribution timeline with a single period. Django admin gains a `ChannelAttribution` inline.
  - **Purge.** `purge_out_of_target_messages` keeps the same channel-level keep-set (any in-target period ∪ `to_inspect` ∪ forward sources) and additionally prunes a kept in-target channel's individual **out-of-period** messages — e.g. those left behind after an analyst narrows a period — while `to_inspect` channels keep every message.
- **Sortable channels list** — the channel list toolbar gains a sort selector (name, first/last activity, followers, messages, organization) with an ascending/descending toggle; sorting runs client-side, within each section, and pushes channels with no value to the end.
- **Genuine Bridging Centrality (Hwang et al. 2008)** — new `BRIDGINGCENTRALITY` measure: betweenness × degree-based bridging coefficient `(1/d(v)) ÷ Σ 1/d(i)`, identifying topological bridges between high-degree regions (no community basis needed). The earlier composite that borrowed the name is renamed **Community Bridging** (measure token `BRIDGING` unchanged, node key `bridging_centrality` → `community_bridging`), and the `bridging` robustness attack relabelled to match — so the two distinct notions of "bridging" no longer share a name.

### Fixes
- **Media-file data loss in `purge_out_of_target_messages`** — on-disk media is stored under paths keyed by the Telegram *file* id (`photos/{id}.ext`, no per-message segment), so a forwarded copy and a kept in-target message reference the **same** file; the purge collected files from the messages it was about to delete *without* checking surviving rows, so purging an out-of-target forward could unlink the media of a kept in-target message. `collect_media_files` is now reference-aware — a file is removed only when no surviving row still points at it (matching `purge_orphan_media`) — and dateless in-target messages are no longer purged.
- **Edge-weight rescaling silently corrupted scale-sensitive analytics** — `build_graph` rescales edge weights to a maximum of 10, which leaked into algorithms that assume the natural weight scale. **Katz centrality** returned invalid (negative) scores because the inflated spectral radius pushed past NetworkX's default `alpha=0.1` (power-iteration diverged and the numpy fallback produced negatives); `alpha` is now derived from a Gershgorin bound on the spectral radius (guaranteed `< 1/λ_max`). **SIR spreading efficiency** saturated (`min(weight, 1.0)` ≈ 1 for most edges) and is now normalised by the max weight. The **Kamada-Kawai layout seed** placed strongly-tied nodes *farther* apart (it reads weight as a distance); weights are now inverted so strong ties pull nodes closer, matching the ForceAtlas2 pass it seeds.
- **Two silent attribution-period bugs** — both bite the new time-bounded model without raising. **(1)** `t.me/` *mention* edges ignored the period cutoff: `network/graph_builder.py` applied `channel_cutoff_q()` to forwards and degree counts but not to the references through-table, so mentions made in a channel's *out-of-period* messages leaked into the graph as edges (and skewed `PARTIAL_REFERENCES` weights, whose denominator *was* period-filtered) — the chokepoint is now applied to mentions too, restoring the documented "only in-period messages contribute" invariant. **(2)** `/manage/` bulk-assign wiped attributions on a bad organization id: `bulk_assign` deleted every selected channel's attribution timeline *before* resolving the organization, so a non-existent (or non-integer) `organization_id` silently erased all attributions and returned `200 OK` — the id is now validated up front, an invalid one is a `400` with no deletion, and only an explicit `null` still clears (intentional unassign).
- **Spreading-efficiency (SIR) robustness attack ignored the run seed** — the `spreading` removal strategy ranked nodes from a hardcoded RNG instead of the configured `--robustness-seed`, so changing the seed left every spreading attack curve and its null-model z-score byte-for-byte identical; the run's seeded generator is now threaded into the SIR scorer.
- **Vacancy `STRUCTURAL_EQUIV` over-scored broad-audience candidates** — its in-amplifier (cosine-in) component computed `sqrt(|V∩C| / |V|)` rather than a true cosine, dropping the candidate's own amplifier-set size; it now computes a proper cosine over each candidate's in-target in-amplifier set, consistent with the cosine-out side, in both the `structural_analysis` export and the channel page's Vacancy Analysis card.
- **`structural_analysis` crashed on the smallest per-year graphs during a `--timeline-step year` run** — the alternative embedding layouts (t-SNE and UMAP, 2D and 3D) carried small-graph fallbacks that under-counted their real minimum, so a per-year subgraph with only a handful of channels aborted the whole export instead of degrading to the Kamada-Kawai fallback. Three distinct failures, all in `network/layout.py`, all reachable directly (no error handling) from the year-export loop in `structural_analysis._run_year_export`: **(1)** the reported one — `tsne_positions_2d` / `tsne_positions_3d` set `perplexity = min(30, max(5, n // 4))`, whose floor of 5 violates scikit-learn's `perplexity < n_samples` rule for `n` in {4, 5} (observed in the wild as `ValueError: perplexity (5) must be less than n_samples (5)` on a 5-node year); perplexity is now additionally capped at `n - 1`. **(2)** `tsne_positions_3d` guarded only `n < 4`, but `_laplacian_features` yields just `n - 2` columns for small graphs and scikit-learn's PCA-based t-SNE init requires `n_components (3) <= n_features`, so a 4-node graph raised `ValueError: n_components=3 must be between 0 and min(n_samples, n_features)=2`; its guard is raised to `n < 5`. **(3)** `umap_positions_2d` / `umap_positions_3d` likewise guarded only `n < 4`, but UMAP's spectral initialiser raises `TypeError: Cannot use scipy.linalg.eigh for sparse A with k >= N` on 4-node graphs; both guards are raised to `n < 5`. All four embedding layouts now fall back to Kamada-Kawai for graphs too small to embed, verified across dense, sparse, and disconnected graphs of 3–30 nodes.
- **Betweenness centrality inverted tie strength** — NetworkX minimises the `weight` argument as a path *distance*, but ours is connection *strength*, so shortest paths skirted the strongest ties and the lightly-trafficked node scored as the broker; `compute_betweenness` now ranks over `distance = 1/weight`, correcting the `BETWEENNESS` and `BRIDGING` measures and the betweenness/bridging robustness attacks.
- **Undirected projections dropped reciprocal-edge weight** — `DiGraph.to_undirected()` keeps only one direction's weight when both `(u, v)` and `(v, u)` exist, so mutual ties lost weight (and which half survived depended on edge order); the new `network.utils.to_undirected_sum` sums both (`W + Wᵀ`) and now feeds Leiden, Leiden-CPM, Walktrap, Louvain, current-flow betweenness, and the algebraic-connectivity stat.
- **Vacancy Analysis card disagreed with its export** — `VacancyAnalysisView` counted deleted (`is_lost`) messages while the `structural_analysis` export filtered them out, so the same vacancy scored differently in the two places; the card's five message queries now use `Message.objects.alive()` to match the export.
- **"Amplifier Jaccard" vacancy score was mislabelled** — it computes `|A ∩ B| / |A|` (coverage/recall of the vacancy's orphaned amplifiers), not a Jaccard; renamed to **Amplifier Coverage** across the UI and docs (dropping the incorrect Small 1973 citation), with the internal `AMPLIFIER_JACCARD` token kept so saved configs keep working.
- **Bridging centrality wasn't the published measure its name implied** — it weighted betweenness by an ad-hoc, unbounded Shannon entropy of neighbour communities; it now uses the bounded `[0, 1]` **participation coefficient** (Guimerà & Amaral 2005), the canonical community-role quantity, keeping the product on betweenness' scale. The robustness bridging attack shares the same helper, so measure and attack stay in lock-step.
- **HITS ignored edge weights** — `nx.hits` treats the graph as binary, so hub/authority scores disagreed with the weight-aware PageRank/Katz/Leiden. A new weighted `compute_hits` (power iteration on the weighted adjacency, normalised to sum 1) now backs the `HITSHUB`/`HITSAUTH` measures and the HITS robustness attacks.
- **Community modularity always used the directed null model** — even for partitions optimised on the undirected projection, so the reported figure didn't match what was optimised. The community tables now report undirected modularity (on the summed `W + Wᵀ` graph) for undirected-basis strategies (Leiden/Louvain/Walktrap/CPM/label-prop/k-core/weak-CC) and directed modularity for the rest, with per-community contributions following the same basis.
- **The robustness null model preserved no real structure** — it fixed the topology and only permuted the weight multiset. It is replaced by a directed weighted **configuration-model** null (`rewire_strength_preserving`): Maslov–Sneppen degree-preserving edge swaps plus iterative proportional fitting preserve each node's in/out degree (exactly) and in/out strength (via IPF) while randomising the wiring, so z-scores isolate higher-order structure rather than the strength sequence the attacks already rank on (existing z-scores change).
- **Vacancy Analysis card defaulted to a narrower window than its export** — the interactive card's fallback `months_after` was 12 while the form, the `structural_analysis` export, and the documentation all used 24; the view fallback is now 24 too.
- **Per-community modularity contributions didn't reconcile with the reported modularity** — the headline figure was weighted but each community's contribution was computed from unweighted edge counts, so the parts didn't sum to the whole; contributions now replicate NetworkX's weighted per-community term and reconcile with the headline.
- **Whole-network Content Originality / Amplification Ratio counted deleted messages** — they ran on all messages while the per-channel `CONTENTORIGINALITY`/`AMPLIFICATION` measures use `Message.objects.alive()`, so the aggregates didn't match the per-channel column; both whole-network figures now use `.alive()` too.
- **Channel "Inbound/Outbound connections" were a non-portable rescaled value** — `in_deg`/`out_deg` summed the ×10/max-normalised edge weight, an arbitrary scale that shifts with the largest edge in each export; they now sum the raw tie weight and are relabelled **In-strength / Out-strength** (the interactive graph panel keeps a true connection count).
- **Freeman centralization was reported for non-centralities** — the network-stats table computed a centralization row for every measure, including audience/activity attributes (followers, message count), local coefficients (Burt's constraint, ego density, local clustering), and behavioural metrics, where score-concentration around the maximum has no centralization meaning; it is now restricted to genuine centrality indices (degree/strength, PageRank, HITS, betweenness, flow betweenness, harmonic, closeness, Katz, bridging).
- **Consensus matrix was contaminated by non-community partitions** — it co-associated channel pairs across every strategy, and its `ORGANIZATION`-exclusion test compared an uppercase name against the lowercase keys in `communities.json`, so it never excluded anything: the manual ORGANIZATION labels and the structural decompositions (weakly/strongly connected components, k-core shells) all inflated agreement (WEAKCC's giant component co-assigns nearly every pair). The consensus now excludes `organization`, `weakcc`, `strongcc`, and `kcore` case-insensitively, leaving only genuine community-detection partitions.
- **Closeness and harmonic centrality ignored edge weights** — they ran on raw hop counts while betweenness (and PageRank/Katz/HITS/flow betweenness) all route on tie strength, so the suite mixed two incompatible notions of distance. Both now compute over the same `distance = 1/weight` proximity projection (Opsahl, Agneessens & Skvoretz 2010) as betweenness, via a shared `proximity_distances` helper; the matching `closeness`/`harmonic` robustness attack scorers were weighted to match. Per-channel closeness/harmonic values change and are no longer bounded to `[0, 1]` (a weighted distance can be below 1), and the two robustness attacks' z-scores shift accordingly.

### Backward incompatibility
- **Recency-weighted edges** — the opt-in recency-decay weighting for graph edges has been removed.
- **Per-channel Top messages panel removed** — the channel detail page's Top messages dropdown (added in [0.22]) and its backing view/route/export read-path are gone; the global `/messages/highlights/` feed and per-message interest scoring are unaffected.

(1) It used `Channel.objects.in_target()` to compute the keep-set, but that queryset additionally filters by `DEFAULT_CHANNEL_TYPES` and excludes `is_lost=True` / `is_private=True`, so any channel marked in-target but currently flagged lost/private or of a wrong type would silently lose all its messages — on the dev DB this was 14,691 messages (24% of the slated deletion) across 187 channels. The new command filters only by the in-target *marker* (`in_target_override=True` or `organization.is_in_target=True`), so transient flags can no longer erase history. (2) The `referenced_ids` query used `.exclude(message_set__forwarded_from__isnull=True)` on a multi-valued relation, which Django translates to "exclude channels that have *any* message with NULL forwarded_from" — i.e., almost every channel. On the dev DB this collapsed 6,639 intended forward-source channels down to 1, so the script was silently deleting messages from channels that *do* feed forwards into the in-target set. The new code uses the correct `.filter(message_set__forwarded_from__isnull=False)`. (3) Django's `FileField`/`ImageField` doesn't delete the underlying file when the row is deleted, so the old script's "to reclaim disk space" claim was half-true — DB rows went away but the `.jpg`/`.mp4` payloads stayed. The new command captures every media-row file path before the cascade delete and removes them from storage afterwards (497 files on the dev DB). End-to-end impact on the dev DB: the buggy script flagged 61,871 messages for deletion; the corrected command flags 675 (the genuinely out-of-target ones) and actually reclaims their on-disk media. Added `--dry-run` to preview, `--yes` / `-y` to skip the interactive prompt, and a refusal-to-proceed when zero channels are marked in-target. Wrapped in `transaction.atomic()`. Test coverage in `webapp/tests.py:PurgeOutOfTargetTests`. Surfaced in Manage → Maintenance as a second panel below the database-optimization one, with a live preview of how many messages and media files would be deleted before the analyst commits.

Pre-computed in Python and stored as `data/structural_similarity.json` (lower-triangle float matrix). Documented in `docs/whole-network-statistics.md` and `docs/export-formats.md`. Also fixes a pre-existing issue where `--html`, `--consensus-matrix`, and any other HTML-only output would silently fail to copy `js/` and `css/` static assets unless `--2dgraph` was also selected.
- **Partition agreement matrix (NMI) in `network_table.html`** — when two or more community strategies are active, a new section shows the Normalised Mutual Information between every pair of strategies as a heatmap matrix. NMI = 1 means the two partitions are identical up to label permutation; NMI = 0 means they are statistically independent. Each pair is computed on the intersection of channels assigned in both strategies, which handles `ORGANIZATION` (which silently skips unassigned channels) correctly. The matrix also appears in `network_table.xlsx`. Formula: NMI(U,V) = 2·I(U;V) / (H(U)+H(V)) (Kvalseth 1987; Fred & Jain 2003). Documented in `docs/whole-network-statistics.md`.
- **`--self-references` flag** — `structural_analysis` gains a `--self-references` / `--no-self-references` flag (default: off). When on, cases where a channel forwards from or mentions itself are included as self-loop edges and contribute to edge weights and centrality measures. Off by default because self-loops carry no network-structural information and have no effect on most centrality measures. Available as an **Include self-references** checkbox in the Edge weights fieldset of the Operations panel.
- **Self-reference filter on Forwards received tab** — the *Forwards received* tab on channel detail pages gains a select control with three options: *Include self-references* (default), *Do not include self-references*, and *Show only self-references*. A self-reference occurs when a channel re-posts its own content, making it appear among its own forwards received. The selection is preserved across pagination.
- **Self-loop rendering in 3D graph** — when `--self-references` is on, self-loop edges were previously invisible in `graph3d.html` because the Bézier control point collapsed to the node position. They now render as a small arch above the node, sized proportionally to the node's radius (`arm = 1× radius`, `peak = 3.5× radius`).
- **Custom and sticker reactions tallied under a `"custom"` bucket** — `_save_reactions()` previously skipped custom-emoji and sticker reactions entirely, understating total reaction counts on channels whose communities use custom emoji packs. All non-standard reactions are now aggregated into a single `MessageReaction` row with `emoji="custom"` and the summed count. Plain emoji reactions are stored individually as before.
- **Separate forward and mention edge weights in GEXF, GraphML, and CSV exports** — `weight_forwards` and `weight_mentions` are now stored as distinct edge attributes alongside the combined `weight` in all three export formats. Both carry the raw forward/mention counts (before any normalisation) from which `weight` is derived, letting analysts distinguish deliberate reposting (forwards) from inline link citations (mentions) in downstream tools such as Gephi, R/igraph, and pandas.
- **CSV export** (`--csv`) — `structural_analysis` gains a `--csv` flag that writes two files alongside the existing outputs: `nodes.csv` (one row per channel, same columns as `channel_table.xlsx` — label, URL, subscribers, messages, in/out-degree, all computed measures, community assignments, activity period) and `edges.csv` (five columns: `source_label`, `target_label`, `weight`, `weight_forwards`, `weight_mentions`, where `weight_forwards` and `weight_mentions` are the raw forward/mention counts before normalisation). Available as a **CSV files** checkbox in the Operations panel alongside the GEXF and GraphML options.
- **Label propagation** (`LABELPROPAGATION`) — new community detection strategy. Uses the semi-synchronous label propagation algorithm (Cordasco & Gargano 2010 / Raghavan et al. 2007): each node starts with a unique label and iteratively adopts the majority label of its neighbours until convergence. Parameter-free, deterministic, and runs in near-linear time — suited as a fast baseline for large graphs where Infomap or MCL are slow. The graph is symmetrised to undirected before running; edge weights are not used.
- **Local clustering coefficient** (`LOCALCLUSTERING`) — new node measure for `structural_analysis`. Computes the directed local clustering coefficient (Fagiolo 2007) for each channel: the fraction of directed triangles through the node relative to all possible directed triads. A high score means the channel is directly embedded in closed citation cycles — mutual amplification loops with specific neighbours. Pairs with `EGODENSITY` to distinguish channels at the edge of a clique (high ego density, low clustering) from active participants in circular amplification (high on both). Nodes with total degree < 2 receive 0.0. Available in `--measures` and the Operations panel Measures grid.
- **Closeness centrality** (`CLOSENESS`) — new node measure for `structural_analysis`. Computes in-closeness centrality for each channel: the Wasserman-Faust normalised score reflecting how easily the rest of the network can reach this channel along citation paths. Channels with high closeness are structurally well-positioned to receive information quickly — they sit at the convergence of many short incoming paths. Isolated nodes and nodes outside the main component receive 0.0. Available in `--measures` and the Operations panel Measures grid.

- **E-I Index and Inter-community Edge Ratio** — two new metrics for evaluating structural competition versus cohesion. The **E-I Index** (Krackhardt & Stern 1988) is computed per community as *(external − internal) / (external + internal)*; range −1 (fully cohesive) to +1 (fully competitive). A weighted **Mean E-I Index** summarises the whole network's balance. The **Inter-community Edge Ratio** is the raw fraction of all directed edges that cross community boundaries. Both appear in the strategy table in `network_table.html` and `network_table.xlsx`; E-I Index also appears as a column in `community_table.html` and `community_table.xlsx`. Documented in `ANALYSIS.md`.
- **Network cohesion metrics** — four new academically validated measures appear in a dedicated *Cohesion* group in the Network Statistics table and XLSX: **Transitivity** (global clustering coefficient, fraction of closed triads; Luce & Perry 1949 / Watts & Strogatz 1998), **Global Efficiency** (mean reciprocal directed path length over all node pairs, handles disconnected graphs gracefully; Latora & Marchiori 2001), **Algebraic Connectivity** (Fiedler value λ₂ of the graph Laplacian, zero for disconnected graphs, larger for more robust networks; Fiedler 1973), and **In/Out-degree CV** (coefficient of variation of the degree distributions, quantifies hub concentration; Pastor-Satorras & Vespignani 2001). All four are documented in `ANALYSIS.md`.
- **Directed Avg Path Length and Directed Diameter** — the mean and maximum directed shortest-path distances computed on the largest strongly connected component (SCC), following edge direction. Complement the existing undirected path metrics. A ‡ footnote appears when the SCC is smaller than the full graph.
- **Backoffice admin** (`/manage/`) — new staff-only app replacing Django admin for day-to-day data management. Powered by a Django REST Framework JSON API (`/manage/api/`). Six sections: **Channels** (searchable/filterable table with inline org assignment, group chip management, and bulk assign/group operations for hundreds of channels at once), **Organizations** (inline CRUD with color picker and is_interesting toggle), **Groups** (inline CRUD), **Search Terms** (add-by-Enter list), **Events** (event types + events with date and type filters), **Users** (create/edit/delete Django users; email is used as the username). A *Manage* button appears in the top navigation for staff users.
- **Message reactions** — emoji reactions are now collected for every message during crawling (initial fetch and stats refresh) and stored in a new `MessageReaction` model. Per-message reactions appear as inline chips in the channel detail message list. A channel-level breakdown of the top 10 emoji reactions by total count is shown just above the stats graphs, and a *Total reactions* summary card is added when reactions are present.
- **Selectable whole-network stat groups** (`--network-stat-groups` / *Network stat groups* in the Operations panel): the seven groups of whole-network structural metrics (`SIZE`, `PATHS`, `COHESION`, `COMPONENTS`, `DEGCORRELATION`, `CENTRALIZATION`, `CONTENT`) can now be selected independently, mirroring how `--measures` and `--community-strategies` work. Default is `ALL`. Each group gates both the computation and the table output; deselecting `PATHS` (path lengths, reciprocity, clustering) and `COHESION` (global efficiency, algebraic connectivity) skips the expensive O(n·m) BFS and eigendecomposition steps on large networks.
- **ChannelGroup model** — channels can now be assigned to one or more named groups via Django admin. Groups appear as checkbox filters in the Get Channels and Export Network Operations panel cards; selecting one or more groups restricts the run to channels in those groups. CLI flags `--channel-groups` (comma-separated) are added to both `crawl_channels` and `export_network`.
- **Ad-hoc search terms in the Operations panel**: the Search Channels card now has an *Extra search terms* textarea (one term per line). Terms entered there are searched alongside the database terms for the current run only. ## [0.13] - 2026-04-20
*New community detection algorithms. Consensus matrix. Events timeline.*

### New features
- `LEIDEN_CPM_COARSE` and `LEIDEN_CPM_FINE`: Leiden optimisation with the **Constant Potts Model** (CPM) objective (Traag et al. 2011). Unlike modularity, CPM has no resolution limit — communities are defined as groups whose internal edge density exceeds a resolution parameter γ. The two variants differ only in their default γ (`--leiden-coarse-resolution`, default 0.01; `--leiden-fine-resolution`, default 0.05). Lower γ gives few, large communities; higher γ gives more, smaller ones. Both symmetrise the graph to undirected (same as `LEIDEN`).
- `MCL`: **Markov Clustering** (van Dongen 2000). Alternates matrix expansion (random-walk diffusion) and inflation (contrast amplification) until convergence. Works natively on the directed weighted graph without symmetrisation. The inflation parameter r controls granularity (`--mcl-inflation`, default 2.0; typical range 1.5–4.0).
- `INFOMAP_MEMORY`: **Second-order (memory) Infomap** (Rosvall et al., Nature Communications 2014). Builds a state network of directed-edge contexts — each state node represents "currently at channel B, having arrived from channel A" — and minimises the map equation on it. Captures sequential forwarding patterns invisible to first-order Infomap. Source nodes receive virtual entry states so they participate in the flow.
- `WALKTRAP`: **Walktrap** (Pons & Latapy 2006). Computes short random-walk distances between channels (4 steps) and applies Ward's agglomerative clustering, producing a full dendrogram cut at the modularity-maximising partition. Symmetrises to undirected.
- All four strategies are available in the Operations panel strategy selector and included in the `ALL` shortcut. The three tunable parameters (CPM resolutions, MCL inflation) are exposed as numeric inputs in the Operations panel Computation section.
- **Consensus matrix** (`consensus_matrix.html`): a new standalone page showing a lower-triangle balloon plot where each cell represents a channel pair. The balloon size grows and the colour shifts from blue to red as more non-ORGANIZATION community detection strategies agree in placing the pair in the same community. Channels are sorted by plurality community assignment then by name. A legend shows one circle per agreement level; hovering a cell shows a tooltip. Generated only when `export_network --consensus-matrix` is passed (or the matching checkbox in the Operations panel is ticked). Requires at least one non-ORGANIZATION strategy.
- `data/meta.json` now includes `has_consensus_matrix` (bool) and `community_distribution_threshold` (int). When `has_consensus_matrix` is true, the Community Statistics page injects a **Consensus matrix** button into its navigation bar automatically.
- **Event markers on home-page charts**: each time-series chart on the home page now draws a dashed vertical line for every event whose date falls within the chart's time span. The line color matches the EventType color. Hovering the line shows a popup listing all events in that month (date, action type, subject).
- **Configurable organization × community distribution threshold** (`--community-distribution-threshold N`, default 10): communities below this percentage in every organization row are hidden from the cross-tab tables. The threshold is stored in `data/meta.json` and read by the JS at page load, so changing it on re-export is reflected without editing any HTML. Previously hard-coded at 5 %; the new default is 10 %. The value can also be set from the Operations panel (Distribution threshold % field).

### Fixes
- **`ChannelCrawler`: `is_private` not cleared on normal crawl completion**: the non-rate-limited completion path set `is_lost=False` but omitted `is_private=False`, leaving a stale `is_private=True` on channels that had previously been private but were later made accessible. The rate-limited early-exit paths already cleared both flags; the normal path now does the same.
- **`ChannelCrawler`: `is_lost` not cleared when a seed is identified as a user account**: when the `ValueError` fallback determined a numeric seed was a user account, only `is_user_account=True` was set. If the channel had previously been marked `is_lost=True` (e.g. from an earlier failed lookup), that flag persisted. Now `is_lost=False` is set alongside `is_user_account=True`, consistent with the private-channel paths.

A "Normalize axes [0–1] per network" toggle min-max scales each network's values independently, making size-dependent measures (degree, fans, message count) directly comparable across networks of different sizes.
  - `graph/index.html` gains a "Compare network" section listing all copied `_2` files and linking to the comparison page.

### Improvements
- `search_channels` now prints progress and results (was fully silent): each search term with found/new counts, and a summary on completion.
- `crawl_channels` and `export_network` now use colour to distinguish section headers (cyan) from step detail lines (plain), warnings (yellow), and final success (green).
- New `GRAPH_OUTPUT_DIR` option sets the directory where `export_network` writes all output files (default: `graph`). Relative paths are resolved from the project root. When the Django development server is running, the output is also served at `http://localhost:8000/graph/`, so a separate HTTP server is no longer needed for local preview.
- Various performance improvements across the backend and frontend.

### Fixes
- `telegram_location` was silently discarded on every crawl: the field was assigned from the Telegram API response but missing from `update_fields` in `set_more_channel_details`, so location data was never persisted.
- `has_been_pinned` was never updated by `--refresh-messages-stats`: the refresh path uses `QuerySet.update()`, which bypasses `Message.save()` where `has_been_pinned` is set. Messages that first became pinned after their initial crawl would show `pinned=True` after a refresh but `has_been_pinned=False`, losing the historical record once they were unpinned. The refresh now explicitly sets `has_been_pinned=True` when the Telegram API reports a message as currently pinned.
- A message that both forwards from channel B and contains a `t.me/B` link (common when editors include inline attribution) was counted in both `forwarded_counts` and `reference_counts` in the graph builder, doubling that edge's contribution to the weight. References to a channel that is already the `forwarded_from` source of the same message are now excluded from `reference_counts`.
- `refresh_degrees()` for interesting channels only counted forwards toward their in/out-degree totals, while `refresh_cited_degree()` for non-interesting channels counted both forwards and `t.me/` references. This made the same field mean different things depending on whether a channel was interesting. Both paths now count forwards and references, matching the edge construction in the graph builder.
- `refresh_degrees()` did not respect `REVERSED_EDGES`: it always stored "cited by" in `in_degree` and "cites" in `out_degree` regardless of the setting, while `refresh_cited_degree()` correctly swapped the two fields when `REVERSED_EDGES=False`. The two paths are now consistent.
- When `DRAW_DEAD_LEAVES=True` with `REVERSED_EDGES=False`, no dead leaves were ever drawn: `refresh_cited_degree()` stores citations in `out_degree` in that configuration, but the dead-leaves inclusion filter always checked `in_degree__gt=0`. The filter now checks the correct field for the active edge direction.
- Dead-leaf nodes with no citations in the active date window appeared as isolated ghost nodes when `DRAW_DEAD_LEAVES=True` was combined with `--startdate`/`--enddate`: they were selected based on their all-time cached degree rather than the date-filtered edge set. Dead leaves that end up with no edges after date filtering are now removed from the graph and the output tables.
- Amplification factor only counted forwards from channels with `is_interesting=True`, but when `DRAW_DEAD_LEAVES=True` the graph includes edges from dead-leaf channels, so a channel heavily forwarded by dead leaves showed low amplification while appearing well-connected in the graph. The measure now counts forwards from all channels present in the graph, keeping it consistent with the edge structure.


## [0.7] - 2026-03-23
*Widening the selection of whole-network measures. Adding more node measures and comparing them.*

### New features
- Three new node measures for political research, all available in `NETWORK_MEASURES` and included in `ALL`:
  - `BURTCONSTRAINT` — Burt's constraint (structural hole brokerage; low = cross-community broker)

- Whole-network metrics moved out of `community_table` into a dedicated `network_table.html` / `network_table.xlsx`; `community_table` now contains only per-community rows.
- `network_table.html` now features an interactive scatter plot where any two measures can be compared on log-log axes, with a power-law trend line. The pair of measures is selected dynamically via dropdowns.
- `export_network` now always generates `graph/index.html`: a landing page listing every available output file (map, channel table, network table, community table, XLSX downloads) with a short description of each and links to the relevant documentation sections. The page is generated regardless of `--nograph` or `--table-format`.

### Improvements
- Graph mini-site no longer depends on jQuery; all DOM interactions rewritten in vanilla JS.
- Stats charts (messages, views, forwards, subscribers, avg involvement) are now rendered client-side with Chart.js instead of Bokeh, using a filled line style better suited to time-series data. Django views return JSON; the browser draws the charts. Removes the `bokeh` dependency.
- Graph export data is now split into typed JSON files under `graph/data/`: `channels.json` (per-node metadata, measures, and community assignments), `channel_position.json` (spatial layout and edges), `communities.json` (strategy definitions and per-community metrics), `network_metrics.json` (whole-network metrics and modularity). The graph mini-site and all HTML tables read these files at load time rather than having data baked in at export time.
- All three HTML tables (`channel_table.html`, `network_table.html`, `community_table.html`) are now static shells that load and render data client-side from `graph/data/*.json`; the HTML files themselves never need to be regenerated — only the data files change between exports.
- Graph mini-site assets reorganized into subdirectories: `graph.html` (was `index.html`), `css/graph.css`, `css/tables.css`, `js/graph.js`, `js/tables_sort.js`, `js/channel_table.js`, `js/community_table.js`, `js/network_table.js`.

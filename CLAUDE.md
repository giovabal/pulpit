## Working rules

- **NEVER run `git add`, `git commit`, or `git push` unless the user explicitly requests that exact operation in their message.** Each operation requires its own explicit instruction: "commit" authorises a commit only; "push" authorises a push only; "commit and push" authorises both. After finishing any code change ŌĆö however large or small ŌĆö stop completely. Do not commit. Do not push. Do not revert and push. Wait for the user to send a separate message.
- Committing and pushing directly to the `main` branch is fine ŌĆö no need to create a feature branch first when the user asks to commit or push (still only do so when explicitly requested, per the rule above).
- Run `ruff check . --fix && ruff format .` before declaring any code change done.
- Smoke-test changes with a quick `python -c "..."` call where practical before finishing.
- When you touch any first-party JavaScript (`webapp_engine/map/js/`, `backoffice/static/`, `webapp/static/`), run `npm run lint:js` (ESLint flat config in `eslint.config.mjs`; requires `npm install` once). Inline `<script>` JS in Django templates is not in scope of this linter.
- When you touch the static-export HTML files (`webapp_engine/map/*.html`), run `npm run lint:html` to catch accessibility regressions (requires `npm install` once). Django templates are not in scope of this linter.
- `npm run lint` runs both JS and HTML linters.
- CHANGELOG entries should be informative but short: one tight paragraph (or a few bullets for larger changes) that names *what* shipped and *why it matters*, not a feature-by-feature walkthrough. Avoid long flag inventories and configuration dumps ŌĆö link to the relevant docs instead.

## Architecture

**Pulpit** crawls Telegram channels, analyzes their network relationships, and generates an interactive force-directed graph visualization.

### Data flow

1. User adds `SearchTerm` entries in Django admin
2. Operations panel (`/operations/`) or `search_channels` finds channels via Telegram API ŌåÆ `Channel` records
3. User labels channels over time via `ChannelLabel` periods in the `/manage/` channel editor; labels live in `LabelGroup` axes (a *partition* group holds Ōēż1 label per channel at a time; exactly one group is *primary*), are defined under `/manage/labels/`, and a channel is *in-target* during any period whose `Label` has `is_in_target=True`
4. Operations panel or `crawl_channels` fetches messages and resolves cross-channel references
5. Operations panel or `structural_analysis` builds the graph, detects communities, runs layout, writes output to `graph/`

Its blocks are *citation-role / structural-equivalence classes* (Lorrain & White 1971), **not** cohesive communities ŌĆö it recovers disassortative / core-periphery / bipartite source-amplifier structure modularity is blind to; seeded for reproducibility. Besides `mode=FLAT|NESTED` it takes two more per-instance enum params (empty default = off, so bare-token keys stay stable): `weights=POISSON|EXPONENTIAL` fits the **weighted** SBM (Peixoto 2018) with edge weights as covariates ŌĆö POISSON needs integer weights (pair with `TOTAL`), EXPONENTIAL positive reals (`PARTIAL_*`); validated before the graph-tool import; unset = binary fit, invariant to `--edge-weight-strategy`. `refine=MCMC` runs multiflip-MCMC posterior sampling (`SBM_MCMC_WAIT`/`SBM_MCMC_SAMPLES`) and reports each node's max-marginal block (Peixoto 2021 `PartitionModeState`) plus a per-channel **assignment-confidence** companion column (`sbm_confidence` + instance suffix via `sbm_confidence_key`; flows to channels.json/CSV/XLSX/GEXF/GraphML and the channel table like `module_role`). ### Network measures

Configured via `--measures` on `structural_analysis` (comma-separated).

| Key | Description |
| :-- | :---------- |
| `PAGERANK` | PageRank score (default) |
| `HITSHUB` | HITS hub score |
| `HITSAUTH` | HITS authority score |
| `INDEGCENTRALITY` | Freeman-normalised in-degree centrality (Freeman 1978; Wasserman & Faust 1994 ┬¦5): `deg_in(v) / (nŌłÆ1)`, the count of *distinct* citing channels normalised by the star-graph maximum. **Unweighted** ŌĆö `nx.in_degree_centrality` discards edge weights, so the ranking is invariant to `--edge-weight-strategy`; the weighted counterpart is the **In-strength** column `in_deg`. Feeds Freeman centralisation in `community_stats` (the star bound is exact for it) |
| `OUTDEGCENTRALITY` | Freeman-normalised out-degree centrality (Freeman 1978; Wasserman & Faust 1994 ┬¦5): `deg_out(v) / (nŌłÆ1)`, the count of *distinct* cited channels normalised by the out-star maximum ŌĆö the *expansiveness* dual of `INDEGCENTRALITY`. **Unweighted** ŌĆö `nx.out_degree_centrality` discards edge weights, so the ranking is invariant to `--edge-weight-strategy`; the weighted counterpart is the **Out-strength** column `out_deg`. Feeds Freeman centralisation in `community_stats` (the star bound is exact for it) |
| `BURTCONSTRAINT` | Burt's constraint (Burt 1992, 2004): `c(v) = ╬Ż_w (p_vw + ╬Ż_q p_vq ┬Ę p_qw)┬▓` with p the row-normalised mutual weight; low = structural-hole broker, high = embedded in redundant ego-network. Typically in [0, 1] (theoretical max Ōēł 1.125, Burt 1992 ch. 2). `nx.constraint` symmetrises direction internally (`mutual_weight(u,v) = w(uŌåÆv) + w(vŌåÆu)`, `N(v) = preds Ōł¬ succs`), so the score is **direction-invariant** ŌĆö `--edge-weight-strategy` matters, the as-built citation orientation does not. `null` for isolated nodes |
| `LOCALCLUSTERING` | Directed local clustering coefficient (Fagiolo 2007): `c^D(u) = T^D(u) / [2 ┬Ę (d^tot ┬Ę (d^tot ŌłÆ 1) ŌłÆ 2 d^Ōåö)]`, the count of directed triangles through `u` summed over the four pattern types (cycle, middleman, in-triangle, out-triangle) divided by the maximum allowed by `u`'s degree configuration. In `[0, 1]`; 0 for isolated nodes and nodes with total degree < 2. Pulpit calls `nx.clustering(graph)` *without* a `weight` argument, so it is **unweighted** ŌĆö `--edge-weight-strategy` does not affect the ranking. Fagiolo's formula sums all 8 directed triangle orientations symmetrically, so the score is also **direction-invariant** (same value on `G` and `G.reverse()`), like `BURTCONSTRAINT` |
| `RECIPROCITY` | Node-level reciprocity (Garlaschelli & Loffredo 2004): twice the number of mutual partners (`pred Ōł® succ`) over the number of in- plus out-partners, self-loops excluded ŌĆö the share of a channel's citation partners that are mutual, the node-level rung of the whole-network / per-community Reciprocity ladder. In `[0, 1]`; `null` for isolated nodes (Burt-constraint convention; hand-rolled because `nx.reciprocity` raises on the isolated in-target nodes Pulpit keeps). **Unweighted** (invariant to `--edge-weight-strategy`) and **direction-invariant**. Dead leaves score 0.0 by construction (their out-citations are out of scope) |
| `MODULEROLE(basis=STRATEGY)` | Guimer├Ā & Amaral (2005) within-module role: emits numeric `within_module_z` (within-community degree z-score, unweighted) **and** numeric `participation` (the participation coefficient ŌĆö weighted evenness of ties across communities, the continuous bridging score the role labels quantise) + categorical `module_role` label (7 roles from the z/P plane). Dispatched specially (needs a community partition); `basis` is per-instance ŌĆö an explicit value (any strategy, incl. a `LABELGROUP<id>` partition), else blank/auto-resolved (prefers `leiden_directed`, then first available). `module_role` is **not** in `measures_labels` ŌĆö it is exported as a string column like `organization` (channels.json node_keys, CSV, XLSX, a "Role" column in `channel_table.js`); `within_module_z` and `participation` flow the normal numeric measure path |
| `AMPLIFICATION` | Forwards received from in-target channels / own message count |
| `CONTENTORIGINALITY` | 1 ŌłÆ (forwarded messages / total messages); `null` if no messages |
| `DIFFUSIONLAG(window=DAYS)` | Median hours from original post date to forward date (within a reaction `window`, default 30 days; `0` disables); `null` for channels with no dated forwards; low = early adopter, high = late amplifier. A bare `DIFFUSIONLAG` inherits `--diffusion-window` |
| `ALL` | All of the above, each parameterised measure once with default parameters |

**Measure tokens & parameters.** `--measures` is an *ordered* comma-separated list parsed by `network.measures.parse_measures` into `MeasureInstance` objects. Two measures take parameters ŌĆö `DIFFUSIONLAG(window=ŌĆ”)` and `MODULEROLE(basis=ŌĆ”)` (keyword args; the legacy positional form `MODULEROLE(STRATEGY)` still parses) ŌĆö and may appear **more than once** with different parameters; the others are drop-once. Every parameterised-measure node-attribute key is **parameter-suffixed** (`diffusion_lag_window_90`, `within_module_z_basis_leiden_directed`, `participation_basis_leiden_directed`, plus the role companion `module_role_*`) so repeated instances never collide. `network.measures.canonical_measure_key` strips the suffix back to the base (used by `CENTRALITY_/BEHAVIOURAL_MEASURE_KEYS` matching and `channel_table.js` column grouping); `role_companions` derives a role measure's categorical companion key from its numeric column. The Operations panel selects measures via a drag-and-drop builder (palette ŌåÆ ordered drop-zone, parameterised chips repeatable with inline parameter inputs); the suffixing/canonicalisation logic is mirrored in `webapp_engine/map/js/channel_table.js`.

**Community-strategy tokens.** Community strategies use the **same machinery**: `--community-strategies` is parsed by `network.community.parse_strategies` into `StrategyInstance` objects. `LEIDEN_CPM(resolution=ŌĆ”)` takes parameters and may repeat (e.g. a multi-resolution CPM scan); the rest are drop-once. Each instance's partition is stored under its suffixed `node['communities']` key and surfaces as a parameter-suffixed community column in CSV/GEXF/GraphML; `canonical_strategy_key` / `strategy_display_label` (Python) and `strategy_label` (`webapp_engine/map/js/labels.js`) strip/label it. The shared token grammar/parsing/suffixing lives in `network/tokens.py`; the Operations-panel drag-and-drop builder is the shared `runner/static/runner/js/token_builder.js`, instantiated once for measures and once for community strategies. A measure's community `basis` names a strategy **family** (e.g. `LEIDEN_CPM`) and resolves to the first selected instance of that family at compute time. CPM resolution now lives in the per-instance token, not in `[computation]` (a loader migration folds the old keys + `LEIDEN_CPM_COARSE/FINE` presets into tokens).

### Edge construction

- `Message.forwarded_from` ŌĆö channel whose content was forwarded
- `Message.references` ŌĆö channels mentioned via `t.me/[username]`

Edge weight has four strategies (`--edge-weight-strategy`): `NONE` (unweighted ŌĆö every citation worth 1), `TOTAL` (raw forwards+references count), `PARTIAL_MESSAGES` (count divided by the *amplifier*'s total messages ŌĆö what fraction of everything the citing channel posted was about the cited one), and the default `PARTIAL_REFERENCES` (count divided by the *amplifier*'s citing messages ŌĆö the ones carrying a `forwarded_from` or a `t.me/` reference, so the amplifier's purely original posts don't dilute the denominator). **Direction is fixed: a forward of source Y's content by amplifier X produces an XŌåÆY edge (the citation convention ŌĆö citing ŌåÆ cited). `Channel.in_degree` therefore counts how many channels cite/forward this one (audience); `out_degree` counts how many channels it cites/forwards (curatorial activity). `BURTCONSTRAINT` symmetrises direction internally (Burt's framework is direction-agnostic); `LOCALCLUSTERING` (Fagiolo 2007) keeps the directed graph but its formula sums all 8 directed triangle orientations symmetrically, so the score is direction-invariant in practice (same value on `G` and `G.reverse()`); `MODULEROLE` is likewise direction-invariant ŌĆö within-module degree and participation both count neighbours on both sides (predecessors Ōł¬ successors), so direction enters only through a directional basis partition; the undirected-basis community strategies (LEIDEN, LEIDEN_CPM, LOUVAIN, KCORE) symmetrise to W+WßĄĆ and discard direction (CONSENSUS inherits whatever its input partitions did); every other graph measure (PageRank, HITS, in/out-degree, vacancy PPR) and the directed strategies (LEIDEN_DIRECTED, SBM) treat edges in the as-built citation direction.**

Only messages dated **within a channel's in-target label periods** contribute ŌĆö the single chokepoint is `network/utils.channel_cutoff_q()` (a period-aware `Q(Exists(ChannelLabel ŌĆ”, label__is_in_target=True))`), with `channel_period_date_q(channel)` as the cheap single-channel variant. A graph node's representative label per partition group (its `LABELGROUP<id>` community) is the label whose periods cover the most days inside the analysis window, tie-broken by earliest start (`network/graph_builder.resolve_window_label`); the **primary** group's resolved label additionally drives node colour and the "Label" column (node attribute `organization`) in tables/CSV/GEXF/GraphML. `to_inspect` channels are crawled in full regardless of periods but enter the graph only as dead leaves.

The files mostly just pre-populate the panel forms ŌĆö the keys that also reach runtime are `[scope].channel_types` (ŌåÆ `DEFAULT_CHANNEL_TYPES`: the web app's in-target scope and both commands' `--channel-types` default), `[edges].weight_strategy` (ŌåÆ `--edge-weight-strategy` default), and the `[graph]` cosmetics (palette, dead-leaves colour, output dir).

Key options:
- `[graph]` in `.operations-structural` ŌĆö `community_palette` (committed baseline `vaporwave` with `community_palette_reversed = true`, so the most-vivid colours land on the largest communities; factory default `""`).

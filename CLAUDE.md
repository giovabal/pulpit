# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Working rules

- **NEVER run `git add`, `git commit`, or `git push` unless the user explicitly requests that exact operation in their message.** Each operation requires its own explicit instruction: "commit" authorises a commit only; "push" authorises a push only; "commit and push" authorises both. After finishing any code change — however large or small — stop completely. Do not commit. Do not push. Do not revert and push. Wait for the user to send a separate message.
- Committing and pushing directly to the `main` branch is fine — no need to create a feature branch first when the user asks to commit or push (still only do so when explicitly requested, per the rule above).
- Run `ruff check . --fix && ruff format .` before declaring any code change done.
- Smoke-test changes with a quick `python -c "..."` call where practical before finishing.
- When you touch any first-party JavaScript (`webapp_engine/map/js/`, `backoffice/static/`, `webapp/static/`), run `npm run lint:js` (ESLint flat config in `eslint.config.mjs`; requires `npm install` once). Inline `<script>` JS in Django templates is not in scope of this linter.
- When you touch the static-export HTML files (`webapp_engine/map/*.html`), run `npm run lint:html` to catch accessibility regressions (requires `npm install` once). Django templates are not in scope of this linter.
- `npm run lint` runs both JS and HTML linters.
- CHANGELOG entries should be informative but short: one tight paragraph (or a few bullets for larger changes) that names *what* shipped and *why it matters*, not a feature-by-feature walkthrough. Avoid long flag inventories and configuration dumps — link to the relevant docs instead.

## Architecture

**Pulpit** crawls Telegram channels, analyzes their network relationships, and generates an interactive force-directed graph visualization.

### Data flow

1. User adds `SearchTerm` entries in Django admin
2. Operations panel (`/operations/`) or `search_channels` finds channels via Telegram API → `Channel` records
3. User labels channels over time via `ChannelLabel` periods in the `/manage/` channel editor; labels live in `LabelGroup` axes (a *partition* group holds ≤1 label per channel at a time; exactly one group is *primary*), are defined under `/manage/labels/`, and a channel is *in-target* during any period whose `Label` has `is_in_target=True`
4. Operations panel or `crawl_channels` fetches messages and resolves cross-channel references
5. Operations panel or `structural_analysis` builds the graph, detects communities, runs layout, writes output to `graph/`

- **`network/graph_builder.py`** — Builds the NetworkX `DiGraph` from Django ORM objects.
- **`network/measures/`** — All centrality and influence measures; `apply_*` functions split across `_centrality.py` and `_content.py`; registry in `_registry.py`.
- **`network/community.py`** — Community detection: LEIDEN, LEIDEN_DIRECTED, LEIDEN_CPM, LEIDEN_TEMPORAL, LOUVAIN, KCORE, SBM, SBM_ASSORTATIVE, CONSENSUS — plus the analyst's own partition `LabelGroup`s, each offered as a `LABELGROUP<id>` strategy (the migrated "Organization" baseline; there is no single `ORGANIZATION` token any more — `labelgroup_strategy_tokens()` enumerates the partition-group tokens, `labelgroup_display_labels()` maps them to names, `detect_labelgroup()` builds the partition). (The flow / random-walk detectors INFOMAP, INFOMAP_MEMORY, MCL, WALKTRAP, STRONGCC were removed under the same one-degree rationale as the path/flow measures: modularity is a single-step walk = one degree, while those integrate many steps. LABELPROPAGATION was removed in v0.27 — unweighted *and* undirected, superseded by CONSENSUS as the cheap baseline; the loader migration drops the token from old configs.) **CONSENSUS** (`detect_consensus`, Lancichinetti & Fortunato 2012, `threshold=τ` default 0.5, repeatable per τ) is the consensus partition of the other selected algorithmic strategies: co-assignment fraction ≥ τ → weighted agreement graph → seeded Leiden; single pass (deterministic inputs make the L&F iteration a fixed point). Inputs are the `consensus_eligible()` instances — non-metadata, non-KCORE, non-CONSENSUS (the single source of that rule, shared with the consensus-matrix validation and mirrored by `_consensusExcluded` in consensus_matrix.js, whose balloon plot excludes CONSENSUS to avoid double-counting); needs ≥2 of them (validated in the command and the panel). Dispatched in a second phase of `_compute_communities` after the direct detections (it cannot run through `detect()`, which raises); in `UNDIRECTED_BASIS_STRATEGIES`, and it *joins* the ARI/AMI/NMI/VI partition-comparison matrices. **LEIDEN_TEMPORAL** (`detect_leiden_temporal`, Mucha et al. 2010 multislice CPM via `leidenalg.find_partition_temporal`; params `resolution` γ + `interslice` ω, repeatable) couples the per-year timeline slices with identity links, yielding one **global community-id space** (+ one palette) across years; requires `--timeline-step year` (validated in `_resolve_options` and the panel), is in `EXCLUDED_FROM_ALL` (so `ALL` never implies it), *not* consensus-eligible (its full-range column is a **plurality-across-slices** summary, latest-year tiebreak, applied on the full-range pass; per-year passes get their slice's map), and precomputed once by `Command._compute_temporal_partitions` (per-year `build_graph` with the same args as `_run_year_export` via the shared `_timeline_year_range`/`_clamp_year_window` helpers, backbone-filtered when set) then applied by lookup in `_compute_communities(temporal_results=…, year=…)`. **SBM_ASSORTATIVE** (`detect_sbm_assortative`, Zhang & Peixoto 2020 Bayesian planted partition via graph-tool `PPBlockState`; greedy `multiflip_mcmc_sweep(beta=inf, niter=PP_GREEDY_NITER)`) gives statistically supported *cohesive* communities — the inferential counterpart of LEIDEN — on the undirected W+Wᵀ projection, unweighted, seeded; optional `refine=MCMC` reuses the SBM posterior machinery and writes an `Assortative SBM confidence` companion (`sbm_confidence_key` handles both families; `sbm_confidence_display_label` is canonical-aware); consensus-eligible and in `UNDIRECTED_BASIS_STRATEGIES`. **LOUVAIN** (`detect_louvain`, Blondel et al. 2008) is the classic modularity baseline, re-added unselected-by-default purely for comparability with the large body of older studies that report Louvain partitions — it optimises the same modularity objective on the same symmetrised W+Wᵀ graph as LEIDEN, which supersedes it (Leiden adds the connectivity-guarantee refinement); so it is in `UNDIRECTED_BASIS_STRATEGIES`, a genuine partition (joins the partition-comparison/consensus matrices, *not* in `_COMPARISON_EXCLUDED_STRATEGIES`), parameter-free, and `seed=0`-reproducible. Prefer LEIDEN / LEIDEN_DIRECTED for new analyses. `LEIDEN_CPM` (resolution γ) and `SBM` (`mode=FLAT|NESTED`) are **parameterised** and may be requested more than once with different parameters; `parse_strategies` turns `--community-strategies` into `StrategyInstance` objects (mirroring measures), each mapping to a parameter-suffixed partition key (`StrategyInstance.key`, e.g. `leiden_cpm_resolution_0_05`, `sbm_mode_nested`; just `leiden_directed` for parameter-free strategies). `canonical_strategy_key` strips the suffix; `strategy_display_label` renders the human label. (`LEIDEN_CPM` replaces the old fixed `LEIDEN_CPM_COARSE` / `LEIDEN_CPM_FINE` presets.) **SBM** is a directed, degree-corrected Bayesian stochastic block model (Karrer & Newman 2011; Peixoto 2014/2017) fitted via `graph-tool` (`detect_sbm`, lazy import → clear `ValueError` if the package is absent; it is **not** pip-installable — conda-forge / apt only). Its blocks are *citation-role / structural-equivalence classes* (Lorrain & White 1971), **not** cohesive communities — it recovers disassortative / core-periphery / bipartite source-amplifier structure modularity is blind to; seeded for reproducibility. Besides `mode=FLAT|NESTED` it takes two more per-instance enum params (empty default = off, so bare-token keys stay stable): `weights=POISSON|EXPONENTIAL` fits the **weighted** SBM (Peixoto 2018) with edge weights as covariates — POISSON needs integer weights (pair with `TOTAL`), EXPONENTIAL positive reals (`PARTIAL_*`); validated before the graph-tool import; unset = binary fit, invariant to `--edge-weight-strategy`. `refine=MCMC` runs multiflip-MCMC posterior sampling (`SBM_MCMC_WAIT`/`SBM_MCMC_SAMPLES`) and reports each node's max-marginal block (Peixoto 2021 `PartitionModeState`) plus a per-channel **assignment-confidence** companion column (`sbm_confidence` + instance suffix via `sbm_confidence_key`; flows to channels.json/CSV/XLSX/GEXF/GraphML and the channel table like `module_role`). SBM is directed, so it is *not* in `UNDIRECTED_BASIS_STRATEGIES` (reported with directed modularity) and *not* in `community_stats._COMPARISON_EXCLUDED_STRATEGIES` (it is a genuine partition, unlike the KCORE shell decomposition, so it joins the partition-comparison/consensus matrices — ARI, AMI, NMI, VI). **`--community-backbone-alpha α`** (config `[communities].backbone_alpha`, panel "Community backbone α") runs all *algorithmic* detections on the `disparity_filter` backbone (Serrano et al. 2009; same filter as robustness) instead of the full graph — label groups, measures, layout, and exports stay on the full graph; `_compute_communities` returns `(strategy_results, detection_graph)` and `compute_community_metrics(detection_graph=…)` reports modularity against the backbone the partitions were optimised on.

the export adds the batch-only `TEMPORAL`), neighbour-set equivalence (the binary Ochiai cosine of the in/out neighbour sets, averaged; labelled "Neighbour-set Equivalence" — distinct from the *weighted* Lorrain & White 1971 structural-equivalence matrix), brokerage-overlap Jaccard of the (source-org, amplifier-org) pairs a channel spans — the cross-product of the orgs it directly forwards from and the orgs that amplify it, so a **one-degree structural-position** overlap (same source/audience ecosystems), *not* a claim of content flow between those orgs through it; operationalising the brokerage *concept* of Gould & Fernandez 1989, *not* their flow-based census; labelled "Brokerage overlap" — and content continuity (`ORIGIN_OVERLAP`, `_scores_origin`: Ochiai of shared **pre-closure origin messages** — the vacancy's curated + authored content universe vs the candidate's after-window forwards of pre-closure-dated origins, both sides conditioned on origins predating the closure because origins are temporally censored across it; origin identity is the coordination layer's `(forwarded_from, fwd_from_channel_post)` with `fwd_from_date` fallback; carries an `archive_forward_count` of shared origins authored by the vacancy itself — the rebrand tell; identity evidence per the ban-evasion lineage (Niverthi et al. 2022), blind to re-*uploads* which get fresh authorship). The amplifier-, source- and origin-set overlaps are additionally **null-calibrated**: exact hypergeometric tail p-values against a uniform-draw null over the active universe (Tumminello et al. 2011 statistically-validated-networks scheme), BH-adjusted per vacancy (`_hypergeom_sf`/`_bh_adjust`), surfaced as q columns with significance stars. `ChannelVacancy.successor` records the analyst-labelled **known successor** (Manage → Vacancies): its row is starred, per-measure ranks are reported, and the export payload carries a `validation` block (hits@1/3/5 + MRR across labelled vacancies). The primary-group label ("organisation") of forwarded-from channels is resolved **as of each forward's date** via `ChannelLabel.build_cache`/`label_at` (label membership is time-bounded). Results are returned sorted by first activity date and rendered in a client-side sortable table; **all candidates are always scored** — `only_after_vacancy` filters the card's response rows after scoring, so scores and q-values are identical to the batch export's regardless of the toggle. The interactive card and the structural-analysis export share one scorer — both call `network.vacancy_analysis._scores_abc` / `_scores_origin` (and `_shift_months`) on `Message.objects.alive()` — so they agree by construction. All message queries (orphaned-amplifier detection, candidate selection, and the scorers, in both the card and the export) are **period-aware** via `channel_cutoff_q()`: a message counts only when its channel was in an in-target period at the message date, matching the graph pipeline's chokepoint.

### Network measures

Configured via `--measures` on `structural_analysis` (comma-separated).

| Key | Description |
| :-- | :---------- |
| `PAGERANK` | PageRank score (default) |
| `HITSHUB` | HITS hub score |
| `HITSAUTH` | HITS authority score |
| `INDEGCENTRALITY` | Freeman-normalised in-degree centrality (Freeman 1978; Wasserman & Faust 1994 §5): `deg_in(v) / (n−1)`, the count of *distinct* citing channels normalised by the star-graph maximum. **Unweighted** — `nx.in_degree_centrality` discards edge weights, so the ranking is invariant to `--edge-weight-strategy`; the weighted counterpart is the **In-strength** column `in_deg`. Feeds Freeman centralisation in `community_stats` (the star bound is exact for it) |
| `OUTDEGCENTRALITY` | Freeman-normalised out-degree centrality (Freeman 1978; Wasserman & Faust 1994 §5): `deg_out(v) / (n−1)`, the count of *distinct* cited channels normalised by the out-star maximum — the *expansiveness* dual of `INDEGCENTRALITY`. **Unweighted** — `nx.out_degree_centrality` discards edge weights, so the ranking is invariant to `--edge-weight-strategy`; the weighted counterpart is the **Out-strength** column `out_deg`. Feeds Freeman centralisation in `community_stats` (the star bound is exact for it) |
| `BURTCONSTRAINT` | Burt's constraint (Burt 1992, 2004): `c(v) = Σ_w (p_vw + Σ_q p_vq · p_qw)²` with p the row-normalised mutual weight; low = structural-hole broker, high = embedded in redundant ego-network. Typically in [0, 1] (theoretical max ≈ 1.125, Burt 1992 ch. 2). `nx.constraint` symmetrises direction internally (`mutual_weight(u,v) = w(u→v) + w(v→u)`, `N(v) = preds ∪ succs`), so the score is **direction-invariant** — `--edge-weight-strategy` matters, the as-built citation orientation does not. `null` for isolated nodes |
| `LOCALCLUSTERING` | Directed local clustering coefficient (Fagiolo 2007): `c^D(u) = T^D(u) / [2 · (d^tot · (d^tot − 1) − 2 d^↔)]`, the count of directed triangles through `u` summed over the four pattern types (cycle, middleman, in-triangle, out-triangle) divided by the maximum allowed by `u`'s degree configuration. In `[0, 1]`; 0 for isolated nodes and nodes with total degree < 2. Pulpit calls `nx.clustering(graph)` *without* a `weight` argument, so it is **unweighted** — `--edge-weight-strategy` does not affect the ranking. Fagiolo's formula sums all 8 directed triangle orientations symmetrically, so the score is also **direction-invariant** (same value on `G` and `G.reverse()`), like `BURTCONSTRAINT` |
| `RECIPROCITY` | Node-level reciprocity (Garlaschelli & Loffredo 2004): twice the number of mutual partners (`pred ∩ succ`) over the number of in- plus out-partners, self-loops excluded — the share of a channel's citation partners that are mutual, the node-level rung of the whole-network / per-community Reciprocity ladder. In `[0, 1]`; `null` for isolated nodes (Burt-constraint convention; hand-rolled because `nx.reciprocity` raises on the isolated in-target nodes Pulpit keeps). **Unweighted** (invariant to `--edge-weight-strategy`) and **direction-invariant**. Dead leaves score 0.0 by construction (their out-citations are out of scope) |
| `MODULEROLE(basis=STRATEGY)` | Guimerà & Amaral (2005) within-module role: emits numeric `within_module_z` (within-community degree z-score, unweighted) **and** numeric `participation` (the participation coefficient — weighted evenness of ties across communities, the continuous bridging score the role labels quantise) + categorical `module_role` label (7 roles from the z/P plane). Dispatched specially (needs a community partition); `basis` is per-instance — an explicit value (any strategy, incl. a `LABELGROUP<id>` partition), else blank/auto-resolved (prefers `leiden_directed`, then first available). `module_role` is **not** in `measures_labels` — it is exported as a string column like `organization` (channels.json node_keys, CSV, XLSX, a "Role" column in `channel_table.js`); `within_module_z` and `participation` flow the normal numeric measure path |
| `AMPLIFICATION` | Forwards received from in-target channels / own message count |
| `CONTENTORIGINALITY` | 1 − (forwarded messages / total messages); `null` if no messages |
| `DIFFUSIONLAG(window=DAYS)` | Median hours from original post date to forward date (within a reaction `window`, default 30 days; `0` disables); `null` for channels with no dated forwards; low = early adopter, high = late amplifier. A bare `DIFFUSIONLAG` inherits `--diffusion-window` |
| `ALL` | All of the above, each parameterised measure once with default parameters |

**Measure tokens & parameters.** `--measures` is an *ordered* comma-separated list parsed by `network.measures.parse_measures` into `MeasureInstance` objects. Two measures take parameters — `DIFFUSIONLAG(window=…)` and `MODULEROLE(basis=…)` (keyword args; the legacy positional form `MODULEROLE(STRATEGY)` still parses) — and may appear **more than once** with different parameters; the others are drop-once. Every parameterised-measure node-attribute key is **parameter-suffixed** (`diffusion_lag_window_90`, `within_module_z_basis_leiden_directed`, `participation_basis_leiden_directed`, plus the role companion `module_role_*`) so repeated instances never collide. `network.measures.canonical_measure_key` strips the suffix back to the base (used by `CENTRALITY_/BEHAVIOURAL_MEASURE_KEYS` matching and `channel_table.js` column grouping); `role_companions` derives a role measure's categorical companion key from its numeric column. The Operations panel selects measures via a drag-and-drop builder (palette → ordered drop-zone, parameterised chips repeatable with inline parameter inputs); the suffixing/canonicalisation logic is mirrored in `webapp_engine/map/js/channel_table.js`.

**Community-strategy tokens.** Community strategies use the **same machinery**: `--community-strategies` is parsed by `network.community.parse_strategies` into `StrategyInstance` objects. `LEIDEN_CPM(resolution=…)` takes parameters and may repeat (e.g. a multi-resolution CPM scan); the rest are drop-once. Each instance's partition is stored under its suffixed `node['communities']` key and surfaces as a parameter-suffixed community column in CSV/GEXF/GraphML; `canonical_strategy_key` / `strategy_display_label` (Python) and `strategy_label` (`webapp_engine/map/js/labels.js`) strip/label it. The shared token grammar/parsing/suffixing lives in `network/tokens.py`; the Operations-panel drag-and-drop builder is the shared `runner/static/runner/js/token_builder.js`, instantiated once for measures and once for community strategies. A measure's community `basis` names a strategy **family** (e.g. `LEIDEN_CPM`) and resolves to the first selected instance of that family at compute time. CPM resolution now lives in the per-instance token, not in `[computation]` (a loader migration folds the old keys + `LEIDEN_CPM_COARSE/FINE` presets into tokens).

### Edge construction

- `Message.forwarded_from` — channel whose content was forwarded
- `Message.references` — channels mentioned via `t.me/[username]`

Edge weight has four strategies (`--edge-weight-strategy`): `NONE` (unweighted — every citation worth 1), `TOTAL` (raw forwards+references count), `PARTIAL_MESSAGES` (count divided by the *amplifier*'s total messages — what fraction of everything the citing channel posted was about the cited one), and the default `PARTIAL_REFERENCES` (count divided by the *amplifier*'s citing messages — the ones carrying a `forwarded_from` or a `t.me/` reference, so the amplifier's purely original posts don't dilute the denominator). **Direction is fixed: a forward of source Y's content by amplifier X produces an X→Y edge (the citation convention — citing → cited). `Channel.in_degree` therefore counts how many channels cite/forward this one (audience); `out_degree` counts how many channels it cites/forwards (curatorial activity). `BURTCONSTRAINT` symmetrises direction internally (Burt's framework is direction-agnostic); `LOCALCLUSTERING` (Fagiolo 2007) keeps the directed graph but its formula sums all 8 directed triangle orientations symmetrically, so the score is direction-invariant in practice (same value on `G` and `G.reverse()`); `MODULEROLE` is likewise direction-invariant — within-module degree and participation both count neighbours on both sides (predecessors ∪ successors), so direction enters only through a directional basis partition; the undirected-basis community strategies (LEIDEN, LEIDEN_CPM, LOUVAIN, KCORE) symmetrise to W+Wᵀ and discard direction (CONSENSUS inherits whatever its input partitions did); every other graph measure (PageRank, HITS, in/out-degree) and the directed strategies (LEIDEN_DIRECTED, SBM) treat edges in the as-built citation direction.**

Only messages dated **within a channel's in-target label periods** contribute — the single chokepoint is `network/utils.channel_cutoff_q()` (a period-aware `Q(Exists(ChannelLabel …, label__is_in_target=True))`), with `channel_period_date_q(channel)` as the cheap single-channel variant. A graph node's representative label per partition group (its `LABELGROUP<id>` community) is the label whose periods cover the most days inside the analysis window, tie-broken by earliest start (`network/graph_builder.resolve_window_label`); the **primary** group's resolved label additionally drives node colour and the "Label" column (node attribute `organization`) in tables/CSV/GEXF/GraphML. `to_inspect` channels are crawled in full regardless of periods but enter the graph only as dead leaves.

### Configuration

Configuration is split across four files:

| File | Content | Gitignored | Example | Format |
|:-----|:--------|:----------:|:-------:|:-------|
| `configuration/.env` | Credentials + deployment (Telegram creds, DB, secret key, web access, locale) | ✓ | `configuration/env.example` | KEY=value |
| `configuration/.operations-crawl` | Committed "Pulpit defaults" baseline for the Crawl Channels form | ✗ | — | TOML |
| `configuration/.operations-structural` | Committed "Pulpit defaults" baseline for the Structural Analysis form | ✗ | — | TOML |
| `configuration/.operations-{crawl,structural}-<timestamp>` | Named snapshots written by **Save as defaults** | ✓ | — | TOML |
| `.system` (repo root) | `APP_VERSION`, `REPOSITORY_URL` — managed by the project, do not edit | ✗ | — | KEY=value |

Required (in `configuration/.env`): `TELEGRAM_API_ID`, `TELEGRAM_API_HASH`, `TELEGRAM_PHONE_NUMBER`.

The bare `.operations-*` baselines are committed in git and never written by the app: **Save as defaults** in the Operations panel writes a *new* gitignored `.operations-{crawl,structural}-<timestamp>` snapshot (via `tomlkit`, titled through a modal), and **Load defaults** applies the baseline or any snapshot back onto the form client-side (`webapp_engine/config/writer.py:write_baseline` can rewrite a baseline but is reserved for tests and future data migrations — no production caller). Factory-empty defaults in `webapp_engine/config/defaults.py` fill in when a file is missing or omits a key. Each file carries a `[meta]` block (`title`, `pulpit_version`, `generated_at`) so future Django data migrations can detect the writing release and rewrite the file when key names change. The files mostly just pre-populate the panel forms — the keys that also reach runtime are `[scope].channel_types` (→ `DEFAULT_CHANNEL_TYPES`: the web app's in-target scope and both commands' `--channel-types` default), `[edges].weight_strategy` (→ `--edge-weight-strategy` default), and the `[graph]` cosmetics (palette, dead-leaves colour, output dir).

Key options:
- `[graph]` in `.operations-structural` — `community_palette` (committed baseline `vaporwave` with `community_palette_reversed = true`, so the most-vivid colours land on the largest communities; factory default `""`). Colouring by a label group uses each label's own colour; algorithmic strategies take the palette. `dead_leaves_color` (default `#596a64`), `output_dir` (default `graph`).
- `[scope].channel_types` in `.operations-crawl` (default `["CHANNEL"]`) — channel types in scope; becomes the `DEFAULT_CHANNEL_TYPES` setting.
- `[downloads]` in `.operations-crawl` — `images` / `video` / `audio` / `stickers` / `other_media` (each default `false`). Each can be overridden per run with the matching `--download-X` / `--no-download-X` CLI flag, or via the **Media types** sidebar fieldset on the Operations panel (applies to `--get-new-messages`, `--fixholes`, and `--fix-missing-media` — the three operations that fetch messages from Telegram).

Media is dispatched into five disjoint models: `MessagePicture`, `MessageVideo` (with `is_animated` and `is_round` flags for GIFs/animations and round videos), `MessageAudio` (with `is_voice` flag), `MessageSticker` (with `is_animated` flag), and `MessageOtherMedia`. Analysis options (measures, community strategies, etc.) are command-line flags on `crawl_channels` and `structural_analysis`; see [docs/workflow.md](docs/workflow.md).
